# 方形拼贴图库布局设计

## 背景

当前图库采用接近瀑布流的纵向排版，图片尺寸虽然保留了原始比例，但整体视觉节奏偏松，图片之间和图片与容器之间的留白仍然明显，不够接近用户参考图中那种“紧贴边缘、大小错落、整体像一块完整拼贴墙”的观感。

用户希望新的展示页图片区域更像移动端作品页或摄影社区中的拼贴式图库：

- 图片大小可以不一样
- 缩略图统一裁成方形
- 图片之间尽量没有明显空白缝隙
- 图片和外层框之间也尽量压缩留白
- 公开展示页与后台管理页保持同一种展示逻辑

## 目标

- 将图库区域改为更紧密的方形拼贴布局
- 保持缩略图大小错落，但每个展示单元本身是方形
- 尽量消除图片之间和图片与容器之间的空白感
- 公开展示页与后台管理页共用同一套布局语言
- 保留现有点击看大图、下载、后台编辑按钮等行为

## 非目标

- 不改变国家章节结构和国家介绍区域
- 不改动灯箱功能、下载功能、后台编辑功能本身
- 不引入第三方前端布局库
- 不做人工逐张排版模板编辑

## 方案选择

本次采用用户确认的 `A. 方形拼贴网格` 方案。

选择理由：

- 最接近参考图的“方形块拼贴”效果
- 响应式控制稳定，手机和桌面都容易统一视觉语言
- 可以通过纯 CSS Grid 完成，不增加运行时复杂度
- 后续上传新图片时不需要额外维护模板或人工排序

## 视觉方向

图库区域采用“紧密拼贴式展墙”方向：

- 所有缩略图以方形块呈现
- 方形块大小不完全一致，而是通过不同网格跨度形成节奏
- 间隙极小，只保留细缝，避免图片彼此粘连但不形成明显留白
- 取消边框感，重点突出连续画面
- 章节内部看起来像一整块图像拼贴，而不是一张张独立卡片

## 布局结构

### 基础单元

每张图在列表里都渲染为一个方形卡片单元：

- 外层单元保持 `aspect-ratio: 1 / 1`
- 图片使用 `object-fit: cover`
- 图片超出部分允许裁切
- 不显示边框

### 大小层次

图库不再使用单纯的多列瀑布流，而改为 Grid 拼贴：

- 默认小块：`1 x 1`
- 横向较大块：`2 x 2`
- 中等块：`2 x 1` 或 `1 x 2`

具体分配原则：

- 通过 `nth-child` 周期规则给部分图片分配更大的网格跨度
- 不依赖图片内容分析，也不要求人工标记
- 让每个章节都自然形成“大图 + 中图 + 小图”的混排节奏

这样能保证：

- 大小有变化
- 仍是规则方形系统
- 图片越多，越能形成连续拼贴感

## 间距与边缘控制

### 图片之间

- 图片块之间的 `gap` 压缩到 `2px`
- 手机上必要时进一步压到 `1px` 或 `2px`
- 不再使用当前较明显的卡片底部间距

### 图片与外框

- 国家章节内图片区域的内边距缩小
- 图库容器尽量贴近章节边缘
- 整体目标是保留极窄的呼吸边，而不是出现明显留白框

## 页面一致性

公开展示页和后台管理页继续保持一致：

- 两边都使用相同的拼贴网格布局
- 两边都显示同样的方形裁切缩略图
- 后台管理页只额外保留编辑按钮覆盖层，不改变底层排版方式

## 交互行为

### 展示页

- 点击任意图片继续打开大图灯箱
- 图片 hover 动效保留，但减弱浮起感，避免破坏拼贴整体性
- 手机端不依赖 hover，直接以密集拼贴方式展示

### 管理页

- 编辑按钮继续叠在图片上方
- 拼贴布局下按钮应保持可见、可点击
- 按钮出现方式要避免大幅推开图片或形成额外留白

## 响应式规则

### 手机端

手机端优先匹配参考图观感：

- 列数更紧凑
- 间隙极小
- 大图块和小图块交替出现
- 保持页面向下滚动时像连续作品墙

### 桌面端

桌面端沿用同样语言，但扩展列数：

- 列数增加
- 同样保留大中小拼贴节奏
- 不退回传统卡片式三列瀑布流

## 技术实现

### 模板层

在图片卡片模板中增加拼贴尺寸标识：

- 为每个图片卡片根据索引生成布局类名或数据属性
- 例如 `tile-large`、`tile-wide`、`tile-tall`、`tile-standard`

模板 responsibility：

- 输出统一的图片按钮结构
- 允许后台和公开页共用同一套卡片样式

### 样式层

核心改动集中在 `static/style.css`：

- 将 `.gallery-grid` 从 `column-count` 瀑布流切换为 `display: grid`
- 为卡片定义统一方形约束
- 定义不同拼贴尺寸的跨行跨列规则
- 压缩缝隙、边距、圆角和 hover 效果

### 后端层

后端不需要新增数据库字段。

若模板使用基于索引的拼贴 class：

- 在 Jinja 渲染循环中按 `loop.index0` 计算模式
- 后台动态渲染若需要同样效果，则在前端 JS 中为动态插入卡片同步应用同样规则

## 风险与约束

- 由于统一裁成方形，部分纵向或横向构图会被裁切，这属于本次视觉方向的预期取舍
- 若某一章节图片数量极少，拼贴节奏会比大量图片时更弱，但仍应保持整洁
- 后台编辑按钮在更紧密布局中可能更容易遮挡图片，需要控制按钮尺寸与透明度

## 验收标准

- 页面图片缩略图统一为方形
- 图片块大小有明显层次变化，不是完全等大网格
- 图片之间空隙极小，不再出现明显卡片式留白
- 图片区域与章节边缘之间的空白明显减少
- 公开展示页与后台管理页布局保持一致
- 点击图片查看大图功能仍可正常使用
- 后台编辑按钮仍可正常使用
- 手机端视觉效果接近参考图的拼贴感

## 测试要求

至少覆盖以下验证：

- 页面 HTML 中存在新的拼贴类名或数据属性
- 公共展示页和后台管理页都输出同一套拼贴结构
- 图片仍然使用预览图 URL 和原图下载 URL
- 原有灯箱和后台编辑相关测试不应回归失败
