# 国家章节短介绍与详细导览设计

## 背景

当前公开展示页和管理页中的国家章节只使用单段 `description` 文案。随着国家介绍内容变得更完整，主界面阅读密度过高，国家标题区显得冗长，不利于用户快速扫览不同国家章节。

现有国家介绍生成也采用“整段重写”思路。每次上传新图片后，AI 会重新根据国家样本图生成整段介绍。这种方式虽然能得到完整文案，但会让已经稳定的章节风格频繁波动，也不符合“上传新图后在原有章节文案基础上增量修订”的使用预期。

本设计把国家介绍拆分为“主界面短介绍 + 可展开详细导览”两层，并将 AI 生成逻辑调整为基于“旧文案 + 新图片”的增量改写。

## 目标

- 在主界面压缩国家章节文字密度，只保留短介绍
- 提供同章节内展开的详细导览，不跳页、不弹窗
- 让 AI 一次生成两段文案：`short_description` 与 `long_description`
- 每次上传图片后，仅对受影响国家执行增量改写
- Gemini 失败、超时或额度耗尽时，保留原有国家介绍
- 保持公开展示页与后台管理页的国家介绍结构一致

## 非目标

- 不引入独立国家详情页
- 不引入弹窗式详细导览
- 不对所有国家在每次上传时做全量重写
- 不改变现有国家分组、照片瀑布流和灯箱查看逻辑

## 展示结构

每个国家章节继续保留以下基本结构：

- 国家编号
- 国家名称
- 作品数量
- 国家介绍区域
- 照片瀑布流

国家介绍区域拆为两层：

- 短介绍：默认显示在国家名下方，作为章节题记
- 详细导览：默认隐藏，点击按钮后在同一章节标题区内展开

交互规则：

- 默认状态仅显示 `short_description`
- 展开按钮文案为 `展开导览`
- 展开后按钮切换为 `收起导览`
- `long_description` 展开位置固定在短介绍下方、照片瀑布流上方
- 不跳页，不弹窗，不滚动到其他位置
- 手机端同样默认折叠，展开后仍停留在当前国家章节内部

管理页与展示页应保持相同的国家介绍呈现结构，避免管理员查看时和公开页出现两套逻辑。

## 文案长度规范

### 短介绍

- 用途：主界面常驻显示
- 字数目标：`22-40` 个汉字
- 功能定位：一句高密度的章节题记，只提炼整组照片最核心的气质

### 详细导览

- 用途：点击展开后显示
- 字数目标：`180-320` 个汉字
- 功能定位：展开地点、建筑、自然与文化线索，形成更完整的章节导言

## AI 输出格式

Gemini 每次生成国家介绍时，必须返回双字段 JSON：

```json
{
  "short_description": "一句短题记",
  "long_description": "一段完整导览文字"
}
```

生成要求：

- 优先识别照片中可高置信度确认的城市、地标或景点名称
- 若无法高置信度确认，不允许硬猜，应改用稳妥的场景或建筑类型描述
- 必须概括地点类型与场景线索，如旧城、湖岸、峡湾、教堂、山谷、城墙、港口等
- 必须从国家层面提炼建筑、自然与文化气质
- 文风应保持展览导言感，避免旅游攻略口吻和空泛辞藻
- 不得编造看不出来的事实，不得出现模糊猜测式表达

分工规则：

- `short_description` 只承担章节总气质摘要
- `long_description` 承担展开叙述

## 增量改写策略

### 核心原则

国家介绍不再在每次上传时从零重写，而是在原有文案基础上做增量修订。

Gemini 输入来源调整为：

- 当前国家现有的 `short_description`
- 当前国家现有的 `long_description`
- 这次新上传到该国家的照片

Gemini 任务是：

- 保留旧文案的整体风格与叙述主线
- 根据新图补充、修正或替换地点、景观、建筑与文化线索
- 同时产出新的 `short_description` 与 `long_description`

### 受影响国家更新规则

- 每次上传后，仅重写本次上传涉及到的国家
- 如果一次上传涉及多个国家，分别逐个国家执行改写
- 未涉及到的国家保持原介绍不变

### 首次生成

如果某国家当前没有任何介绍：

- 允许 Gemini 从零生成 `short_description` 与 `long_description`

### 主题大变动场景

增量改写默认偏保守，适合稳定扩展同一国家章节内容。

如果某国家后续上传内容跨度很大，导致章节主题发生明显迁移，例如：

- 原来集中于湖镇与山地风光
- 后续大量新增首都街景、博物馆或宫殿建筑

则应保留一个后续可扩展能力：

- 支持手动触发“完整重写国家介绍”

本次设计先不要求在界面上完成这个入口，但实现结构应允许未来扩展。

## 数据结构变更

现有国家介绍元数据为单字段：

```json
{
  "奥地利": "单段旧介绍"
}
```

目标结构改为：

```json
{
  "奥地利": {
    "short_description": "湖山与旧镇在静光中相互映照。",
    "long_description": "完整章节导览文字……"
  }
}
```

要求：

- 存储层需要支持读取和写入双字段结构
- 公开页与管理页模板都读取统一结构
- API 返回的分组数据中应包含短介绍与详细介绍

## 兼容迁移

系统必须兼容当前已经存在的单段 `description` 旧数据。

兼容规则：

- 如果国家介绍还是旧字符串：
  - 先将其视为 `long_description`
  - `short_description` 初始为空
- 页面渲染时：
  - 若 `short_description` 缺失，则提供一个兜底短介绍
  - 若 `long_description` 缺失，则提供一个兜底详细介绍
- 某国家一旦成功经过新的 Gemini 更新，就会正式转换为双字段结构

这样可以避免旧数据在新结构下失效，也不需要人工一次性重录全部国家介绍。

## 失败回退策略

以下场景视为 AI 更新失败：

- `GEMINI_API_KEY` 未配置
- Gemini 超时
- Gemini 免费额度用完
- Gemini 返回格式错误
- Gemini 返回内容为空

失败时要求：

- 图片上传仍然成功
- 当前国家原有 `short_description` 与 `long_description` 保持不变
- 页面继续显示旧内容
- 后台仅提示失败原因

若国家原本没有任何介绍且这次 AI 又失败：

- 主界面短介绍显示简短兜底文案
- 详细介绍区域显示“导览文字稍后补充”

## 前端交互细节

### 公共展示页

- 国家标题下显示短介绍
- 详细导览折叠在同一章节内
- 展开后样式应像章节导言，而不是普通帮助文本
- 详细导览区域与照片瀑布流之间应有清晰分隔

### 管理页

- 呈现结构与展示页一致
- 方便管理员上传后立刻检查短介绍与详细导览是否合理

### 手机端

- 默认仍保持折叠状态
- 按钮尺寸、留白与行距适配触控
- 展开内容不应造成过于突兀的布局跳动

## 视觉方向

- 短介绍像章节题记，视觉权重稳但不冗长
- 详细导览像“章节前言层”，应与普通正文说明区分开
- 按钮不能太像默认系统折叠控件，应保持展览页面的语气与克制感

## 后端行为变更

上传流程触发国家介绍更新时：

- 先识别本次上传影响到的国家
- 对每个国家收集：
  - 旧短介绍
  - 旧详细介绍
  - 本次新增图片
- 调用 Gemini 增量改写
- 若成功，更新双字段介绍
- 若失败，保留旧值

同步刷新与编辑资料后触发的国家介绍重算，也应复用同一套结构。

## 测试要求

至少覆盖以下验证：

- 旧字符串介绍可兼容读取
- 页面默认只展示短介绍
- 点击后展开详细导览，且位置正确
- 管理页与展示页结构一致
- Gemini 返回双字段 JSON 时可正常存储
- 上传后仅更新受影响国家
- Gemini 失败时保留旧短介绍与旧详细介绍
- 没有任何旧介绍时，兜底文案可正常展示

## 风险与约束

- Gemini 对地点识别的准确性依赖图像信息与置信度，仍应保持“不硬猜”的约束
- 增量改写比全量重写更稳定，但在章节主题发生大迁移时会相对保守
- 数据结构从单字段迁移到双字段后，模板、API 和存储层必须同步更新，否则会出现前后端不一致

## 实施建议

建议按以下顺序实现：

1. 扩展国家介绍存储结构，兼容旧数据
2. 调整 Gemini 输出格式为双字段
3. 修改页面模板和管理页渲染逻辑
4. 改造上传后国家介绍更新为增量改写
5. 增补测试并验证失败回退
